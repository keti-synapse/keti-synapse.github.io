synapse
The single message standard every component imports — definitions only, no execution.

이기종 멀티로봇 플랫폼의 가장 안쪽 표준 레이어로, Topics·표준 메시지 6종·Enum 카탈로그·Pydantic v2 자료구조·미션 모델·standardize()를 한 곳에 모은다. 철학은 단 하나, "정의만 한다"는 것이다. 실행 로직은 바깥(tom_and_gerri·nexus)이 가지며, 여기를 바꾸면 전체가 흔들리므로 가장 신중히 관리한다.

Message Catalog

표준 메시지 6종

외부로 흐르는 모든 통신은 정확히 6개의 토픽으로 끝난다. 토픽은 절대 raw 문자열로 쓰지 않고 항상 Topic.* 상수를 쓴다.

필드타입설명
msg_info.timestampfloattime.time() 기준 송신 시각
msg_info.uuidstruuid4 — JS 안전을 위해 문자열로 직렬화
msg_info.senderstr송신 로봇의 robot_id
msg_info.receiverstr | None수신 대상. None = 브로드캐스트
topicstr항상 Topic.* 값 — raw 문자열 금지
valuedict토픽별 페이로드

Figure 1. 공통 메시지 봉투. msg_info(timestamp·uuid·sender·receiver) + topic + value 세 부분으로 고정된다. uuid는 JS 호환을 위해 uuid4의 문자열로 직렬화한다.

robot.status
Topic.Robot.STATUS
로봇 전체 상태. RobotInfo(id·sub_id·type·shape·capability·version) + RobotState(base_state·pose·battery…) + MissionInfo 세 묶음. 주기적 "전체 상태" 스냅샷이자 단일 진실 원천(SSOT).
⚠️ raw_status(SDK 원본) ≠ base_state(시스템 종합)를 합치지 않는다. (robot_id, sub_id)로 upsert — 1:N 로봇 대응.
mission.event
Topic.Mission.EVENT
미션 진행 중 "방금 일어난 일". MissionEvent 16종으로 표현. Event를 놓쳐도 Status가 SSOT이므로 복구된다.
Status = 주기적 전체 상태 · Event = 실시간 알림.
mission.trigger
Topic.Mission.TRIGGER
외부에서 미션을 트리거. name + args 구조. 미션 시작, wait_input·wait_teleop 해제 등에 사용.
빌더: build_trigger(sender, name, args=None)
command.capability
Topic.Command.CAPABILITY
Capability 호출. value.name = capability 이름, value.params = 인자. base가 getattr(sub, name)(**params)로 디스패치한다.
✓ 단일 토픽 — command.{capability} 동적 토픽은 만들지 않는다(무한 폭발 방지).
command.teleop
Topic.Command.TELEOP
원격조작 입력을 단일 토픽으로 통일. axes · buttons · poses 세 채널을 함께 담는다. 조이패드와 VR을 하나의 표준으로 흡수.
build_joy() / build_vr()는 이 위에 얹힌 편의 빌더일 뿐, 별도 토픽이 아니다.
help.action
Topic.Help.ACTION
인간 개입 액션. HelpAction 기반(retry · skip · abort · takeover). "실패는 정상, 인간 개입 우선" 원칙의 메시지 표현.
빌더: build_help(sender, action, context=None)
내부 토픽: Topic.Network.SEND/RECV는 tom_and_gerri 내부 버스용으로 synapse에 함께 정의되지만 외부 표준 6종에는 포함되지 않는다. 규칙은 동일하다 — 토픽은 언제나 Topic.*를 통한다.

raw_status ≠ base_state ≠ mission_state
세 차원을 절대 병합하지 않는다. 같은 이름(IDLE·TELEOP)이 서로 다른 차원에 등장해도 의미가 다르다 — raw_status는 SDK 원본, base_state는 시스템 종합, mission_state는 미션 진행이다.
battery_state.percentage / is_charging
Battery 필드명은 percentage·is_charging(그 외 voltage·current·temperature). level/charging이 아니다.
정적 메타도 status 안에
capability·shape 같은 정적 메타데이터도 status에 함께 싣는다. 새 엔티티로 분산하지 않는다.
upsert key = (robot_id, sub_id)
1:N 로봇(한 본체에 여러 서브)을 위해 두 키로 upsert. sub_id가 모듈별 상태를 분리한다.
extension / raw_status = dict
표준에 없는 값은 자유 dict로 확장. 표준을 깨지 않고 로봇별 확장을 허용한다.

Figure 2. robot.status 스키마와 차원 분리 원칙. RobotState의 모든 필드는 optional이며, 누락분은 standardize()가 워닝과 함께 흘려보낸다.

Figure 4. Capability 18종 + 자유 확장. REQUIRED_CAPABILITIES = {stop} — 모든 로봇이 stop()을 구현해야 한다. 로봇별 비표준 동작은 free-form custom_capability로 추가한다.

디스패치 규칙: command.capability의 value.name에 capability 이름이 담기고, base가 getattr(sub, name)(**params)로 호출한다. command.{capability} 같은 동적 토픽은 만들지 않는다 — 토픽 폭발을 막는다.
relocate — pose 보정: 로봇 pose 추정치를 자기 map 프레임에서 (x, y, theta)로 보정한다. 서버 소유의 맵 정렬 T·set_pose와는 엄격히 구분되며 — pose만 바꾼다. pose를 리포트하는 로봇만 광고한다.
software_update — 유지보수: NEXUS 릴레이로 로봇이 스스로 git pull 후 재시작한다(edge-authoritative). BUSY·TELEOP·NEED_HELP·ERROR면 거부한다 — 자가 재시작 위험 가드. N-디바이스 게이트웨이는 프로세스 1개만 갱신하고, 대행 디바이스 중 하나라도 BUSY·ERROR면 거부한다.
Core Building Blocks

핵심 자료구조 · 빌더

개발자가 실제로 import해서 쓰는 골격 — Result, StatusBuilder, standardize(), Topic.

Result — capability 반환 표준: Result는 capability가 반환하는 표준 @dataclass(Pydantic 아님)로, 라이브러리가 정의하고 사용자는 반환만 한다. 필드는 success(bool) · data(dict) · message(str) · error(str|None)이며, Result.ok(message="", **data) 또는 Result.fail(error, message="")로 생성한다.
StatusBuilder — status 메시지 빌더: StatusBuilder(robot_id).build(robot_state, mission_info, robot_info)가 status 메시지 dict를 만든다. 각 입력에 standardize()를 자동 적용하고 robot_id를 보장하며, 누락 필드는 워닝으로 흘려보낸다.
standardize() — Postel's Law(받을 땐 관대, 보낼 땐 표준): normalize(data)는 dict·Pydantic·dataclass·None 등 무엇이든 dict로 바꾼다(never raise). standardize(data, model_class)는 normalize 후 model_validate를 시도하고, 실패하면 문제 필드만 떼어 재검증해 (dict, warnings)를 반환한다 — 절대 예외를 던지지 않는다.
Topic — raw 문자열 금지: 토픽은 항상 Topic.* 상수로 접근한다(예: Topic.Command.CAPABILITY = "command.capability").
Topic.* 상수문자열 값
Topic.Robot.STATUSrobot.status
Topic.Mission.EVENTmission.event
Topic.Mission.TRIGGERmission.trigger
Topic.Command.CAPABILITYcommand.capability
Topic.Command.TELEOPcommand.teleop
Topic.Help.ACTIONhelp.action

자료구조 요약. models.py는 Pydantic v2이며 모든 필드가 optional이다. Pose3D{x,y,z,qx,qy,qz,qw}가 정규형(쿼터니언)이고 from_2d/from_rpy/.yaw/.to_2d 변환을 제공한다. Twist{linear, angular}, Joint{positions(required)·names·velocities·efforts}, NetworkState는 internet/server/operator 각각 Link{is_connected, latency_ms, packet_loss}로 구성된다.

기계판독 wire contract (contract/): 모델·enum·미션을 export_schema.py로 뽑은 JSON Schema(synapse.schema.json, draft 2020-12) + 토픽별 골든 픽스처(byte-stable, drift-guard) + interop.json(WebRTC 공통 코덱 베이스라인 VP8+Opus — SDK는 이를 협상해야 함)을 커밋해 둔다. SDK 코드젠(Phase 6~8)이 git 해시로 이 아티팩트를 핀한다. 버전(0.4.4)은 semver 규칙(D-07)으로 올린다.
Mission Model & Sync

미션 모델 · 시간 동기화

missions.py는 미션 자료구조와 간결 헬퍼를, time_sync.py는 NTP식 오프셋 동기화를 정의한다.

Task 5종 · 미션 헬퍼 6종

capability wait_input wait_teleop notify func
개발자가 작성하는 미션 정의
def delivery_5f_to_1f() -> RobotMission:
    return RobotMission(name="delivery", tasks=[
        cap("undock", "undock"),
        cap("go", "navigate_to_pose",
            x=1.0, y=2.0),
        wait_input("confirm", prompt="도착했나요?"),
        cap("dock", "dock"),
        notify("done", "배달 완료"),
    ])

시간 동기화 · 미션 위임 구조

TimeSync — NTP식 4-타임스탬프: TimeSyncMaster(NEXUS) / TimeSyncSlave(TAG)가 4개의 타임스탬프로 offset과 RTT를 NTP식으로 동기화한다.
미션 위임 구조: FleetMission(다중 로봇 DAG, depends_on) → RobotMission(단일 로봇 Task 시퀀스) → Task(5종). 헬퍼 cap·wait_input·wait_teleop·notify·func·call로 간결하게 정의한다.
Fleet 자료구조: FleetMission{tasks: dict[str, FleetTask]}, FleetTask{robot, mission, depends_on, policy}, ExecutionPolicy.on_failure = stop · continue · retry · help(사람 개입 요청 → NEED_HELP/TELEOP, 무응답 timeout 시 FAILED 폴백).
