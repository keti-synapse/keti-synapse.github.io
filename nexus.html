NEXUS

한 사이트의 플릿 매니저 · WebRTC 시그널링 게이트웨이

단일 사이트의 로봇 플릿을 한곳에서 보는 관제 서버이자, 운영자↔로봇 P2P 연결을 이어주는 WebRTC 시그널링 릴레이. 상태의 원본은 로봇(엣지)이 소유하고, NEXUS는 그 사본(뷰·캐시)만 들고 있는 Edge-Authoritative 설계다.

python -m nexus app.py relay.py registry.py fleet.py persistence.py gateway.py config.py v0.0.1
🎮 체험해보기 — 가상 플릿 관제 ⭐ GitHub

실제 로봇 없이 nexus_ui 스타일 화면에서 직접 조종해 볼 수 있습니다.

Overview

한 사이트를 책임지는 뷰·캐시 서버

NEXUS 한 대 = 한 사이트의 플릿 매니저.상태의 진실은 로봇에 있고, 서버는 그 사본을 모아 보여줄 뿐이다.

NEXUS는 진실의 원본(SSOT)이 아니다. 각 로봇(TOM)이 자기 상태의 권위를 가지며, NEXUS는 WebSocket으로 받은 상태를 메모리에 캐시해 대시보드에 비춰주는 뷰다.

핵심 축은 둘이다. 첫째, 연결된 로봇들의 status를 모아 캐시하고 REST로 노출한다(/api/robots). 둘째, 운영자와 로봇이 직접 만나도록 WebRTC 시그널링을 중계한다(/ws/{room}). 핸드셰이크가 끝나면 미디어·DataChannel은 P2P로 직결되고 NEXUS는 데이터 경로에서 빠진다 — 작은 시그널링 JSON만 오간다.

이 두 축 위에 엣지 원칙을 지키는 얇은 서비스가 얹힌다 — FleetMission 조율(/api/fleet/*), 공간 메타(표준맵·로봇맵 정렬 부착 /api/maps·/api/alignments), 3D 트윈용 로봇 description(URDF) 캐시·서빙(/api/description), 컨트롤플레인 명령 릴레이(/api/command), IoT 알람 집계(/api/alarms). 어느 것도 좌표 계산이나 미션 실행을 대신하지 않는다 — 여전히 부착·중계·모니터링만 한다.

의존성도 좁다. NEXUS가 import하는 것은 공통 표준 synapse 하나뿐이며(메시지 토픽 Topic 사용), tom_and_gerri나 tag_* 같은 로봇 코드는 전혀 import하지 않는다 — 코드 결합도 0, 오직 네트워크 메시지 표준으로만 대화한다.

Edge-Authoritative. 연결 끊김(OFFLINE)조차 NEXUS가 단정하지 않는다./api/robots는 server_time을 함께 내려주고, 클라이언트가 server_time − last_seen > ttl 로 스스로 판단한다.

Stack & Boot

기술 스택과 단일 명령 기동

실시간 제어는 엣지에, 미디어는 P2P에 맡기므로 NEXUS는 IO 중심이다.async·WebSocket·OpenAPI를 기본 제공하는 FastAPI + Uvicorn 한 켤레로 충분하다.

패키지는 여전히 단순하다. __main__.py가 uvicorn.run(create_app())을 띄우고, app.py의 create_app() 팩토리가 라우트와 RelayHub·RobotRegistry를 조립한다. relay.py는 룸 기반 시그널링 릴레이, registry.py는 (robot_id, sub_id) 상태 캐시, fleet.py는 FleetMission DAG 조율기, persistence.py는 SQLite 영속 seam, gateway.py는 콕핏 정적 서빙 + config 주입, config.py는 정적 Pydantic 설정을 담당한다(그 외 ratelimit.py·logging_config.py). npm·webpack 같은 빌드 단계는 없다.

terminal · 단일 프로세스 기동
# synapse(공통 표준)는 editable 설치 — tom_and_gerri/tag 는 설치/그 import 불필요
pip install -e ./synapse
pip install -e ./nexus      # fastapi · uvicorn · websockets · pydantic>=2

# 서버 기동 (기본 http://0.0.0.0:8000)
python -m nexus
python -m nexus --host 0.0.0.0 --port 8000

Figure 2. python -m nexus 한 줄이 전부다. --host/--port는 선택, 기본값은 0.0.0.0:8000.

Signaling Relay & Status Cache

하나의 WS, 세 갈래 처리

단일 엔드포인트 /ws/{room}가 메시지 종류에 따라 캐시 / 브로드캐스트 / 1:1 라우팅으로 갈린다.시그널링 JSON은 해석 없이 원문 그대로 전달된다.

topic == "robot.status"로봇 → NEXUS
→
캐시만 (릴레이 X). registry.upsert_status(value)로 (robot_id, sub_id) 엔트리를 갱신. 대시보드용 사본일 뿐 다른 피어에 전달하지 않는다.
type == "ready" (+ sender)운영자 입장
→
등록 후 브로드캐스트. hub.register_sender(sender, ws)로 sender id를 등록하고, receiver가 없으므로 같은 room의 나를 제외한 모두에게 전파.
offer / answer / candidate (+ receiver)핸드셰이크
→
1:1 유니캐스트. receiver id로 등록된 그 피어에게만 전달. 원문 JSON을 그대로 forward — 재직렬화·해석 없음.

Figure 5. RelayHub.targets() + app WS 핸들러의 분기. receiver가 있으면 그 피어에게만, 없으면 room 전체로. status는 라우팅에서 빠져 캐시로 흡수된다.

한 줄 규칙. 핸들러는 메시지의 봉투(sender / receiver)만 보고 JSON 원문을 그대로 중계한다 — SDP·ICE 페이로드는 들여다보지 않는다.robot.status만은 라우팅 대상에서 빠져 레지스트리에 캐시만 된다.대시보드로의 실시간 push는 별도 /ws/dashboard가 담당한다(DashboardHub).

Data Model & Registry

(robot_id, sub_id)로 캐시하고 robot_id로 묶기

레지스트리는 (robot_id, sub_id) 키로 status를 upsert한다.보통 1 로봇 = 1 base = 1 피어지만, 드물게 물리적으로 분리된 디바이스는 별도 sub로 발행한다.

한 로봇이 여러 sub(또는 분리 디바이스)로 status를 따로 발행해도, /api/robots는 이를 robot_id 기준으로 묶어 한 로봇처럼 보여준다. 묶인 그룹의 last_seen은 sub들 중 가장 최신 시각이다. 드물게 물리적으로 갈라진 경우에는 mobile_01.arm 같은 피어 id로 구분한다. robot_id가 없는 status는 라우팅 불가로 폐기되며(경고 로그), 잘못된 메시지가 서버를 죽이지 않게 한다.

필드설명
robots[]등록된 로봇 배열. 각 항목은 robot_id로 묶인 한 로봇.
robots[].robot_id로봇 식별자 (그룹 키).
robots[].subs{ sub_id: entry } 맵. 각 entry는 robot_info · robot_state · mission_info를 담는다.
robots[].last_seenfloat — 묶인 sub들 중 가장 최신 수신 시각.
server_timefloat — 응답 생성 시각. 클라이언트의 OFFLINE 판정 기준.

Figure 9. /api/robots 응답 구조. 각 sub 엔트리는 robot_info · robot_state · mission_info를 담고, 상위에 server_time이 동봉된다.

OFFLINE은 클라이언트가 판정한다. NEXUS는 단정하지 않고, 클라이언트가 server_time − last_seen > ttl 인지로 스스로 판단한다.

Fleet Orchestration · 구현됨

FleetMission — 다중 로봇 조율 (구현 완료)

다중 로봇 조율이 구현됐다(Phase 3).NEXUS가 FleetTask DAG를 조율해 로봇에 미션을 던지고 robot.status.mission_info로 진행을 추적한다 — 실행 주체는 여전히 로봇(엣지)이다.

구현됐습니다. nexus/fleet.py의 FleetMissionRunner가 FleetTask DAG를 실행하고, /api/fleet/* REST API로 시작·조회·중단한다(단위 테스트 tests/test_fleet.py 포함).

NEXUS는 스케줄링을 조율(orchestrate)한다. FleetMissionRunner가 depends_on 기반 DAG를 풀어, 의존 task가 모두 성공하면 해당 로봇에 mission.trigger를 던진다. 완료 추적은 robot.status.mission_info 폴링 — mission_state가 TERMINAL(SUCCEEDED/FAILED/ABORTED)이 되면 task 종료로 본다(Status가 SSOT). 미션 실행 주체는 로봇(TOM)이고, 데이터 모델 FleetMission은 synapse에 정의된다(NEXUS에 두지 않는다).

한 로봇은 한 번에 한 미션만 수행하므로, 같은 로봇을 대상으로 한 task는 의존이 없어도 자동 직렬화된다. 실패 정책 on_failure(stop · continue · retry · help)와 timeout을 지원하고, 진행상황은 run_view로 대시보드에 broadcast된다. 각 task의 skip/실패에는 기계판독 reason 코드(FleetTaskReason — dep_failed·upstream_stop·trigger_undelivered·fleet_timeout 등)와 사람용 detail이 분리돼 붙어, 제3 UI(Unity 등)도 동일 계약으로 분기·로컬라이즈할 수 있다. 요약하면 — NEXUS는 던지고 지켜보고, 로봇이 실행한다. 점유·실패 복구 같은 실행 책임을 엣지에 남기는 것이 Edge-Authoritative 원칙과 일관된다.

컴포넌트책임 · 구현됨
FleetMissionRunnernexus/fleet.py. depends_on 기반 DAG를 풀어 다중 로봇 미션을 조율(orchestrate). 예: 이동 → 제조 → 전달 → 배송 순서를 의존성으로 묶는다. validate_dag가 의존성 존재·사이클을 사전 검증한다.
/api/fleet/*REST — GET/POST templates(빌더 저장, DAG 검증) · GET/POST missions(인라인 fleet_mission 또는 template+args로 시작) · GET missions/{id} · POST missions/{id}/abort.
FleetMission (데이터 모델)synapse.missions에 정의 — NEXUS가 소유하지 않는다.
NEXUS의 역할mission.trigger로 미션을 던지고, robot.status.mission_info 폴링으로 진행을 모니터링한다(콜백 send·get_mission_info 주입 구조).
로봇 (TOM)의 역할실제 Task 실행 · 점유 · 실패 복구 — 실행 책임은 엣지에 남는다.
