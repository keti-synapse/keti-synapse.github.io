AND

Adaptive Network Daemon

tom_and_gerri 레포의 _and_/ 모듈이다.로봇 내부의 in-process 버스(pypubsub)와 외부 전송 백엔드를 잇는 얇은 어댑터 데몬으로, 제어 로직은 없고 전달만 한다.

네트워크 데몬 WebRTC Transport 독립 pypubsub
Definition & Role

전달만 하는 어댑터 데몬

AND는 “로봇이 바깥과 어떻게 통신하는가”를 한 곳에 가둔다.GERRI(제어 SW)는 전송 수단을 전혀 모른 채 표준 토픽으로만 말하고, AND가 그것을 실제 네트워크로 옮긴다.

Adaptive Network Daemon — synapse 내부 버스와 외부 transport 백엔드를 잇는 레이어다.

AND는 두 조각으로 나뉜다. 브리지(daemon.py의 AdaptiveNetworkDaemon)는 synapse 토픽 Topic.Network.SEND/Topic.Network.RECV와 백엔드를 잇고, 백엔드(backend.py의 NetworkBackend 인터페이스 + 구현)는 실제 전송 수단을 담당한다 — WebRTC(rtc2kng), 테스트용 loopback 등.

내부 버스는 pypubsub(in-process·동기 디스패치)이다. 브리지 자체에는 큐도 스레드도 상태도 없다(인스턴스 상태는 self.backend 하나뿐). 그저 한쪽 토픽에서 받은 것을 다른 쪽으로 넘긴다.

Bridge

내부 버스 ↔ 외부 전송

Base(게이트웨이)가 Topic.Network.SEND로 내보낸 메시지는 backend.send()로 흘러 나가고, 백엔드가 외부에서 받은 메시지는 Topic.Network.RECV로 다시 올라온다.

▲ 외부 수신은 역방향 — backend → Topic.Network.RECV → Base

Figure 1. AND 브리지 양방향 배선. 송신: Base가 Network.SEND 발행 → 브리지가 backend.send() 호출. 수신: backend가 외부 메시지 수신 → 브리지가 Network.RECV로 재발행. 변환·검증 없는 pass-through.

방향흐름
송신Base가 Topic.Network.SEND로 발행 → 브리지가 구독하고 있다가 backend.send(message) 호출 → 외부로 전송
수신backend가 외부 메시지 수신 → 주입된 콜백이 Topic.Network.RECV로 재발행 → Base가 받아 처리
상태 없는 pass-through. 브리지는 메시지를 변환·검증 없이 그대로 넘긴다. 생성 즉시 SEND 구독이 활성화되고 외부 수신 콜백이 백엔드에 주입되며, connect()/disconnect()는 전송 계층의 수립·정리를 백엔드에 위임한다.
Transport-Agnostic

전송은 갈아끼울 수 있다

이것이 AND 존재 이유다.GERRI는 오직 Topic.Network.SEND/RECV하고만 대화하므로, 전송 백엔드를 WebRTC에서 다른 것으로 교체해도 GERRI 제어 로직은 한 줄도 바뀌지 않는다.

Rtc2kngBackend · WebRTC ↔ 교체 가능 ↔ LoopbackBackend · 테스트 MQTT? gRPC?
모두 동일한 NetworkBackend 인터페이스(connect/send/disconnect + _deliver)를 따른다.

Figure 2. 전송 독립. 백엔드만 갈아끼우면 된다(브리지·제어 코드는 불변 — 클린 아키텍처). 현재 백엔드는 robot-first WebRTC(rtc2kng)이며, 단위 테스트는 실제 네트워크 없는 LoopbackBackend로 대체한다.

인터페이스 계약은 단순하다. 구현체는 connect()·send(message)를 채우고, 외부에서 메시지를 받으면 부모가 제공하는 self._deliver(message)를 호출해 브리지로 올린다. disconnect()는 추상 강제가 아닌 선택(no-op 기본) — 연결·전송은 필수, 정리는 선택이라는 의도적 비대칭이다.

rtc2kng

WebRTC 백엔드 — robot-first P2P

현재 전송 백엔드는 _and_/rtc2kng/의 WebRTC 구현이다.로봇이 부팅 시 시그널링에 합류하고, operator의 ready를 받으면 로봇이 offer를 주도한다.단일 asyncio 루프(데몬 스레드)에서 1:N operator 연결을 관리한다.

signaling.py
SignalingClient — WebSocket 시그널링 클라이언트. SDP offer/answer·ICE candidate의 전송 계층만 담당. P2P가 열리면 데이터 경로에서 빠진다.
peer_connection.py
RobotPeerConnection — operator 1명과의 aiortc 연결. ICE 협상 + DataChannel("synapse")로 표준 메시지(JSON) 교환.
rtc2kng_backend.py
Rtc2kngBackend — NetworkBackend 구현. robot-first, 여러 operator와 동시 1:N. asyncio 루프 스레드에서 시그널링·peer 구동.
camera_track.py
CameraSource → VideoStreamTrack 어댑터. 카메라를 WebRTC로 송출하기 위한 변환층(현재 attach 지점만 배선).
operator ready → robot offer → operator answer → candidate 교환 → P2P 직결

Figure 4. robot-first 핸드셰이크. 시그널링이 끝나면 미디어·제어는 P2P로 직결되고 서버는 데이터 흐름에서 빠진다. aiortc는 non-trickle이라 로봇 ICE 후보를 offer SDP에 한 번에 담는다.

Gateway Rule

Sub는 네트워크를 직접 만지지 않는다

AND/외부 토픽에 손댈 수 있는 것은 오직 Base(게이트웨이) 하나뿐이다.SubController는 외부 통신을 전혀 몰라야 한다.

규칙: SubController는 Topic.Network.SEND/RECV 등 외부 토픽을 직접 사용하는 것이 금지된다.Sub는 자기 status를 내부 토픽으로만 발행하고, 외부와의 모든 송수신은 Base가 단일 창구로 중계한다.이렇게 해야 전송 교체가 Sub에 전혀 영향을 주지 않고, AND의 “전달만 하는 데몬” 경계가 유지된다.

즉 의존 방향은 Sub → (내부 토픽) → Base → (Network.SEND/RECV) → AND → backend → 네트워크 한 방향으로 흐른다. Sub가 이 규칙을 어기고 외부 토픽을 직접 건드리면 게이트웨이 정책이 깨지고, 백엔드 교체 시 Sub 코드까지 수정해야 하는 결합이 생긴다.
