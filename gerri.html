Global Extended Remote Robot Interface

The Bundle

TAG = TOM + AND + GERRI

물리 HW(TOM)·외부 통신 데몬(AND)·제어 SW(GERRI) 셋이 모여 한 로봇 분(分)의 묶음을 이룬다. 이 레포 tom_and_gerri는 그 중 AND 구현 + GERRI를 담는다.

Architecture

제어 SW 3계층 — Base / Sub / Mission

각 계층은 명확한 단일 책임을 가진다. Sub는 외부 통신을 전혀 모르고, Base가 외부와의 유일한 창구(게이트웨이)다.

🏗️
Base
BaseControllerCommon · base_controller.py
  • 🔀 외부와의 유일한 창구(게이트웨이). 명령 분배 · sub 관리 · 미션 실행을 담당한다.
  • 📡 내부 토픽 Topic.Robot.STATUS 구독 → robot_id를 채워 외부 Topic.Network.SEND로 내보낸다.
  • 📥 외부 Topic.Network.RECV 수신 → receiver(target) 기준으로 자기/해당 sub에 라우팅한다.
  • 🧩 @property capability — 자기 + 모든 sub의 {"common":[…], "custom":[…]}를 종합한다.
  • 🛑 stop() — 진행 미션을 abort하고 각 sub의 stop()을 호출한다(라우팅 분기 최우선 매칭).
↕ 같은 프로세스 객체 그래프 — sub 메서드 직접 호출 (getattr)
🎮
Sub
SubControllerCommon · sub_controller.py
  • 🤖 Capability 구현 단위. 로봇 SDK를 직접 호출한다. 외부 통신은 모른다.
  • 📊 자기 status는 내부 토픽으로만 발행 — pub.sendMessage(Topic.Robot.STATUS, data=self.get_status()). 외부 토픽 사용 금지.
  • ✅ self.capability(공용 enum) + self.custom_capability(메서드 이름 list)를 선언한다.
  • 🔑 stop()은 모든 로봇이 반드시 구현 — 베이스는 NotImplementedError를 던진다.
↑ capability를 시퀀스로 조합
📜
Mission
MissionRunner · mission/runner.py
  • 🔄 RobotMission(Task 시퀀스) 실행기. base가 dispatch와 sender를 주입한다.
  • 👤 사람 개입 — wait_input / wait_teleop(NEED_HELP → TELEOP) 처리. "실패는 정상, 인간 개입 우선."
  • 🚫 FleetMission(다중 로봇 조율)은 여기 없다 → NEXUS 책임.

Figure 2. 3계층 데이터 흐름. 외부 통신은 오직 Base를 통한다. Sub는 내부 버스(pypubsub)로만 status를 올리고, Base가 게이트웨이에서 robot_id를 채워 외부로 중계한다 — 의존 방향은 안쪽(synapse)을 향한다.

Base Controller

Base — 메시지 게이트웨이

통신 단위 하나당 하나의 Base가 존재한다. __init__(info)는 info["id"]를 요구하며, 이것이 robot_id(외부 식별자)가 된다.

Base는 생성 시 세 토픽을 구독해 게이트웨이를 연다 — 내부 Robot.STATUS(sub→base), Mission.EVENT(runner→base), 외부 Network.RECV(AND→base). 게이트웨이로서 sub의 내부 status에는 robot_id를 채워 외부 Network.SEND로 중계하고, 외부 메시지는 receiver 값으로 자기/해당 sub에 라우팅한다.

멤버책임
robot_idinfo["id"]에서 받은 외부 식별자. 게이트웨이에서 sub status·외부 메시지에 채워진다.
sub_controllers항상 dict. 등록된 Sub들의 모음으로, receiver(sub_id) 기준 라우팅 대상이 된다.
missionsname → RobotMission 카탈로그. NEXUS가 이름으로 트리거한다.
mission_runnerMissionRunner(dispatch=self.dispatch_capability, sender=self.robot_id) — base가 디스패치와 sender를 주입한다.
teleop_handler선택 — command.teleop를 해석하는 핸들러.
@property capabilityself + 모든 sub의 {"common":[…], "custom":[…]}를 종합해 status에 노출한다.
dispatch_capability(capability, params=None, target=None) — capability 이름으로 메서드를 getattr 조회·호출해 Result 반환. target(sub_id) 지정 시 그 sub로(예: 양팔 left/right), 미지정이면 base 자신 → 첫 매칭 sub 순.
STOP은 큐 우회가 아니다. 현재 명령 경로는 pypubsub 동기·단일 스레드라 명령 큐가 없다. STOP은 라우팅 분기에서 가장 먼저 매칭되어 즉시 실행되고, 진행 미션은 abort 플래그로 다음 task 경계에서 ABORTED 된다.
Sub Controller

Sub — Capability 구현 (외부 통신 모름)

로봇 SDK를 직접 호출하는 곳으로, __init__(info)는 info["id"](=sub_id)를 요구한다. status는 get_status()로 만들어 내부 토픽으로만 발행하고, robot_id는 Base가 게이트웨이에서 채운다.

tag_template/sub_controller.py — TemplateSub (모바일)
from gerri import SubControllerCommon
from synapse import Capability, Result

class TemplateSub(SubControllerCommon):
    def __init__(self, info, sdk=None):
        super().__init__(info)
        self._sdk = sdk or _FakeSDK()
        self.capability = [
            Capability.NAVIGATE_TO_POSE,
            Capability.VELOCITY_CTRL,
            Capability.DOCK, Capability.UNDOCK,
            Capability.STOP,            # ← 필수
        ]

    def navigate_to_pose(self, x, y, theta=0.0, timeout_s=30.0) -> Result:
        """좌표로 이동. x,y(미터, map frame) · theta(라디안) · timeout_s(초)."""
        try:
            if self._sdk.move_to(x, y, theta):
                return Result.ok(message="도착", reached=[x, y, theta])
            return Result.fail("not_reached", message="도달 실패")
        except TimeoutError:
            return Result.fail("timeout", message=f"{timeout_s}초 초과")

    def velocity_ctrl(self, linear, angular) -> Result:
        """직접 속도 제어. linear(m/s) · angular(rad/s)."""
        self._sdk.set_velocity(linear, angular)
        return Result.ok()

    def dock(self, charger_id=None) -> Result: return Result.ok(message="도킹")
    def undock(self) -> Result: return Result.ok()

    def stop(self) -> Result:
        """필수 — 로봇을 IDLE 로."""
        self._sdk.halt()
        return Result.ok(message="정지")

Figure 3. 실제 TemplateSub 구현(scaffold). 한국어 docstring에 단위·범위·좌표계를 명시하는 것이 컨벤션이다. 모든 메서드는 synapse Result를 반환하고, stop()은 예외 없이 구현된다.

Convention

Capability 컨벤션 — 이름 == 메서드

capability 값이 곧 메서드 이름이다. Base는 getattr(sub, name)(**params)로 자동 매칭해 호출한다 — 분기(if robot == …) 없이 라우팅이 성립한다.

Common — synapse.Capability 표준
고정 시그니처 공용 동작
synapse.Capability enum으로 정의된 표준 동작. 시그니처가 고정되어 어떤 로봇이든 같은 인터페이스로 부른다.
navigate_to_pose velocity_ctrl cartesian_ctrl gripper_ctrl dock undock stop ← 필수
Custom — 자유 형식
로봇별 고유 동작
self.custom_capability(메서드 이름 문자열 list)로 선언. 시그니처가 자유로워 로봇 특수 기능을 노출한다.
home_position make_americano fold_panel set_led
반환은 언제나 synapse Result다. 성공은 Result.ok(message="", **data), 실패는 Result.fail(error, message="")이다. @property capability는 self + 모든 sub의 것을 {"common":[…], "custom":[…]}로 종합해 status에 노출한다 — 관제 UI는 이 목록만 보고 무엇을 시킬 수 있는지 안다.
Mission System

MissionRunner — Task 5타입 · 상태기계 · 인간 개입

자료구조(Task·RobotMission)는 synapse.missions, 상태/이벤트는 synapse.enums에 있다. Runner는 실행만 하고, mission_state는 runner가 소유한다(edge-authoritative).

_execute(task) — 5가지 Task 타입
capabilitydispatch(task.capability, task.params) → Result
functask.func() 실행 — 테스트·빠른 개발
notify알림 로그만, 흐름은 계속
wait_input계획된 입력 대기 — RUNNING 유지, provider로 값 수신
wait_teleop사람 원격조작 호출 — NEED_HELP → TELEOP → TeleopExitAction

Figure 4. Task 5타입. task.policy.on_failure는 stop(미션 FAILED) / continue(다음 task) / retry(retry_count만큼 재시도)를 결정한다. 실패는 정상으로 취급된다.

MissionState 상태기계 · 인간 개입 흐름
IDLE → RUNNING → NEED_HELP → TELEOP → RUNNING → SUCCEEDED
FAILED ABORTED WAIT_INPUT

wait_teleop 흐름: RUNNING → NEED_HELP → TELEOP → (teleop_provider가 TeleopExitAction 반환) → TELEOP_ENDED → RUNNING. resume_done/resume_retry면 다음으로 진행, abort면 미션 ABORTED. 추가로 오퍼레이터는 언제든 자발적 takeover(HelpAction.TAKEOVER)로 개입할 수 있다.

Figure 5. 상태기계와 인간 개입. 모니터/UI는 status(TELEOP)와 event(TELEOP_STARTED/ENDED)로 operator 개입 구간을 감지한다. STOP은 abort() → 다음 경계에서 ABORTED.

tag_template/missions.py — synapse.missions 헬퍼
from synapse.missions import RobotMission, cap, notify, wait_input

def pickup_with_confirm() -> RobotMission:
    """이동 → 사람 확인(계획된 입력 대기) → 복귀."""
    return RobotMission(name="pickup_with_confirm", description="확인 후 픽업", tasks=[
        cap("go", "navigate_to_pose", x=2.0, y=0.0),
        wait_input("confirm", prompt="집을까요?",
                   buttons={"yes": "예", "no": "아니오"}),
        cap("home", "navigate_to_pose", x=0.0, y=0.0),
    ])

Figure 6. 실제 RobotMission 예시. cap·wait_input·notify 헬퍼로 Task 시퀀스를 간결히 정의한다. 미션은 Base의 self.missions 카탈로그에 등록되어 NEXUS가 트리거할 수 있다.

Teleop Handlers

텔레옵 핸들러 (handlers/) — 선택적

synapse는 표준 입력 인덱스만 정하고, 의미 해석은 핸들러가 한다. 모두 통일 시그니처 handle(axes, buttons, poses, target=None)를 따른다. 핸들러는 dispatch(capability, params)만 알면 된다.

🎮 joy.py — 게임패드
MobileJoyHandler — 왼쪽 스틱=주행(전후/회전) → velocity_ctrl, B=STOP.

ManipulatorJoyHandler — A=home_position, B=STOP, 트리거(L2/R2)=gripper_ctrl(0~100%).

synapse.teleop_mapping의 JoyAxis·JoyButton 인덱스를 읽는다. 입력은 sparse해도 되며 없는 인덱스는 0으로 친다.
🥽 vr.py — VR 컨트롤러
SingleArmVRHandler — poses[PoseChannel.LEFT/RIGHT_HAND](6DOF) → cartesian_ctrl, 아날로그 그립(axes) → gripper_ctrl(0~100%).

양팔(dual-arm)은 arm별 sub 둘. 손→팔 라우팅은 메시지의 target(receiver)으로 결정 — 별도 dual-arm 핸들러가 필요 없다.
Developer Flow

새 로봇 추가하기

표준이 있으니 scaffold를 복사해 SDK만 연결하면 된다. Base는 sub 등록·핸들러·미션 카탈로그만 구성하면, 게이트웨이·STOP·디스패치·미션 실행을 BaseControllerCommon이 제공한다(올인원).

1
scaffold 복사
cp -r gerri/tag_template/ gerri/tag_{myrobot}/
2
SubControllerCommon 상속
self.capability list를 설정하고, capability 메서드에서 로봇 SDK를 호출한다. 반드시 stop() 구현, Result 반환, 네트워크 토픽은 절대 건드리지 않는다.
3
BaseControllerCommon 상속
self.sub_controllers = {"main": MySub(...)}로 등록. 선택적으로 self.teleop_handler = MobileJoyHandler(self.dispatch_capability), 그리고 self.missions 카탈로그 등록.
4
로컬 단독 → NEXUS 연결 테스트
capability 단위 호출 → status 발행 확인 → 미션 시퀀스 실행. NEXUS 없이 로컬 단독으로도 테스트 가능하다.

tag_template/base_controller.py — TemplateBase

올인원 — sub·핸들러·미션만 구성
from gerri import BaseControllerCommon
from gerri.handlers import MobileJoyHandler
from .missions import patrol_demo, pickup_with_confirm
from .sub_controller import TemplateSub

class TemplateBase(BaseControllerCommon):
    def __init__(self, info):
        super().__init__(info)
        # 1:1 — sub_controllers 는 항상 dict
        self.sub_controllers = {
            "main": TemplateSub({
                "id": info["id"],
                "category": info.get("category", "mobile"),
                "model": info.get("model"),
            }),
        }
        # 텔레옵 핸들러(선택) — 게임패드로 주행
        self.teleop_handler = MobileJoyHandler(self.dispatch_capability)
        # 미션 카탈로그
        self.missions = {
            "patrol_demo": patrol_demo(),
            "pickup_with_confirm": pickup_with_confirm(),
        }

COCKPIT · 1:1 Remote Operation

🕹️ COCKPIT — 1:1 원격 조작 UI

단일 로봇과 WebRTC P2P로 직결되는 1:1 정밀 조작 인터페이스다. 카메라·오디오·DataChannel·조이스틱을 한 화면에서 다룬다. 코드는 이 레포의 gerri/interface/cockpit/에 있다 — 별도 컴포넌트가 아니라 GERRI가 제공하는 운전석이다.

MODE A
Standalone (LAN · NEXUS 없이)

TOM의 standalone_server가 직접 정적 파일을 서빙하고, 로컬 /ws/{room} 릴레이로 시그널링한다. 서버 없이 현장에서 바로 로봇에 붙어 조작.

MODE B · 약 99%
NEXUS-context

NEXUS 게이트웨이가 /{site}/{floor}/tag_{robot}/에서 콕핏을 서빙하고, NEXUS /ws/{room}가 시그널링을 중계한다. 대시보드에서 클릭 한 번으로 진입.

Figure 8. 두 가지 호스팅 모드. 어느 쪽이든 제어·미디어는 P2P로 흐르고 서버는 시그널링만 중계한다.

서버는 정적 index.html에 런타임 설정만 문자열 치환으로 주입한다. 빌드된 정적 파일 안의 마커를 window.__COCKPIT__ = {robotId, signalingUrl, mode} 한 줄로 바꿔치기하는 방식이라, 호스팅 모드에 따라 robot_id·시그널링 URL·mode만 달라진다.

같은 프론트, 두 서버. 프론트엔드(index.html · cockpit.js · cockpit.css)는 순수 정적 파일이라 로직을 전혀 갖지 않는다. 주입된 window.__COCKPIT__ = {robotId, signalingUrl, mode} 객체만 읽고 동작하므로, 동일한 정적 파일이 Standalone·NEXUS 양쪽에서 그대로 재사용된다.
RelayHub — 순수 라우팅

targets(room, peer, message)는 전송 계층과 무관한 순수 함수다. 메시지에 receiver가 있으면 그 peer에게만 1:1 전달하고, 없으면(예: "ready") 같은 room의 나머지 전부에게 broadcast한다. 내용은 해석하지 않고 JSON을 그대로 중계한다.

디커플링. NEXUS는 tom_and_gerri/tag 코드를 import하지 않는다. 콕핏 프론트는 데이터(정적 파일)일 뿐 — tag_{robot}/ 폴더가 있으면 그걸 서빙하고, 없으면 기본 콕핏으로 폴백한다. 로봇별로 자기 콕핏을 자유롭게 커스터마이즈할 수 있다.
